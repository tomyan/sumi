# @sumi/lang-sumi

Lezer grammar and CodeMirror 6 language support for the sumi template
language. The grammar parses the sumi template shell — sections, tags,
attributes, control blocks, interpolation — and nests the real Go and CSS
parsers into the language islands via `parseMixed`.

## Usage

```js
import { EditorState } from "@codemirror/state";
import { EditorView } from "@codemirror/view";
import { sumi } from "@sumi/lang-sumi";

new EditorView({
  parent: document.body,
  state: EditorState.create({ doc: "...", extensions: [sumi()] }),
});
```

## Layout

| File               | Role                                                    |
| ------------------ | ------------------------------------------------------- |
| `src/sumi.grammar` | Lezer grammar (source of truth).                        |
| `src/tokens.js`    | External tokenizers for raw sections and brace islands. |
| `src/index.js`     | `LanguageSupport` + `styleTags` + `parseMixed` wiring.  |
| `test/cases.txt`   | Lezer tree tests (standard format).                     |
| `demo/`            | Standalone CM6 demo page.                               |

`src/parser.js` and `src/parser.terms.js` are generated by
`npm run build:parser`.

## Scripts

- `npm run build:parser` — regenerate the parser from the grammar.
- `npm test` — build the parser, run the tree tests and mixed-nesting tests.
- `npm run build:demo` — bundle `demo/demo.js` to `dist/demo.bundle.js`
  (open `demo/demo.html`).

## Mixed-language nesting

The template shell is sumi's own grammar; the islands nest published parsers:

- `<script>` / `<sumi:imports>` bodies → `@codemirror/lang-go` (`goLanguage.parser`)
- `<style>` bodies → `@codemirror/lang-css` (`cssLanguage.parser`)
- `{if}` / `{for}` clauses and every `{expression}` → `goLanguage.parser`

Bare Go expression islands nest cleanly: the Go parser accepts a fragment as
`SourceFile(ExprStatement(...))` and error-tolerantly produces correctly
typed nodes, so highlighting is faithful inside `{expr}` and `{if}`.

## Known gaps

- **`{for}` range clauses** nest as Go but Go has no "range clause" entry
  point, so `range` and the trailing `key=` produce a couple of error nodes.
  Identifiers still colour; only `range`/`key` fall back to plain text.
- **Dotted / prefixed component tags** (`<pkg.Card>`, `<sumi:TextInput>`)
  start lowercase, so they highlight as `ElementName` (element) rather than
  `ComponentName`. Bare capitalised components (`<Card>`) are correct.
- **Text-brace divergence**: text `{expr}` islands are read with balanced
  braces here; the Go template parser stops at the first `}`. Only matters
  for the rare literal `}` inside a text interpolation.
- **Raw sections end at the first `</`**: a `</` inside a Go string literal
  in `<script>` would end the section early (same limitation as lang-html).
- Section delimiters (`<script>` etc.) and the keyword literals are matched
  verbatim, exactly as the Go section splitter does — sections never carry
  attributes.
